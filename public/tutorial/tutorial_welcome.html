Audience

This tutorial assumes no programming experience, and attempts to explain advanced computer science concepts in simple terms. Some familiarity with programming certainly helps, but is not required.

Environment

To start coding, all you need is the Dargon command-line interface (CLI) and a text editor. The CLI is the all-in-one tool for programming in Dargon (it's an interpreter, compiler, and library builder - neat!). If you're not familiar with what a command line is, see this nifty guide.

Running DIR in the command line (./dargon) without any inputs will begin the Dargon Interpreter (DIR). Here, DIR will await your input in what is more commonly referred to as a "Read-Eval-Print-Loop" (REPL):

~/home/demo$ ./dargon
Dargon version 0.1-DEBUG (Linux) - Apr 16 2024, 21:51:07
(C) Kyle Morris 2023-2024 | See LICENSE.txt for license information.

Welcome to the Dargon Interpreter (DIR)! For help, type 'help', to exit type 'quit'.
DIR>

This is great for testing out things, or just messing around with the language.

Hello Dargon!

Let's jump right in starting with an established tradition with our own little twist. Start by creating a new text file called "Hello.dargon":

print("Hello Dargon!")
Hello.dargon

The Dargon Interpreter (DIR) comes with a set of pre-defined functions, one of which is "print". This function takes a string and outputs it to the console. Let's use Dargon to execute this file:

~/home/demo$ ./dargon Hello.dargon
Hello Dargon!
~/home/demo$

Great! But this is not all that Dargon is capable of. Before we go too much further, let's go over Dargon's syntax more thoroughly.

Comments

A "comment" refers to a piece of code that is set to be intentionally ignored, which is useful for documenting your code for yourself and others. In Dargon, there are 3 kinds of comments. Lines that start with the pound ("#") character are line comments, in which the entire line is ignored. If a pair of parenthesis surround two pounds, it creates a block comment that can extend as many lines as you want, given the end is also marked.
The final type of comment is the To-Do comment. Like any other comment, DIR ignores it. However, when prompted, DIR can analyze all todo comments and provide a report to the programmer listing all of the things that still need to be done. This is useful when a To-Do list of your project is requested. See below for some examples:

# This is a line comment, and thus ignored by DIR
Dargon code...

(#
This is a block comment and
extends multiple lines.
Wow!
#)
 Dargon code...

#! This is a todo comment, and will be reported by DIR!
Dargon code...

(#!
Finally this is a todo block comment!
Notice how the matching end does not contain an excalamation mark.
#)

Declarations

In programming languages, a declaration is a statement which declares a constant, variable, function, or some other programming construct. Dargon allows you to declare variables, meaning that it's value can change throughout the code, or constants, meaning that its value cannot change once initialized. Create a new file named "Declare.dargon" and plug in the following lines:

fixed real PI = 3.1415926
int errorCode = 0
bool flag # Given the default value 'false'
print("Pi up to 7 digits is equivalent to: {PI}")
Declare.dargon

This file so far declares two things: a constant (fixed) real number named "PI", and a variable integer named "errorCode". It also assigns each one a value. It then prints the contents of "PI" to the screen by inserting its value into the string via the curly brace syntax (described later). Run this via DIR:

~/home/demo$ ./dargon Declare.dargon
Pi up to 7 digits is equivalent to: 3.1415926
~/home/demo$

Dargon is statically-typed, meaning the type of each variable must be given on declaration (Dargon does not "guess" what the type is). In this case, the types are real and int respectively.
When something is declared, if it is not given a value, it uses its default value. For integers and reals, this is 0 and 0.0 respectively, for booleans this is false, and for strings this is the empty string (""). Everything always has a value in Dargon - no nulls are allowed (unless it's of a nullable type - more on that later).

In Dargon, the rule of thumb is: if you, the programmer, cannot deduce the type of a declaration, then Dargon cannot and will not allow it.

Dargon also allows multiple declarations per line. You can get away with not declaring types if the same type applies to each variable. See this example:

# Notice how each one is separated with a comma ','
fixed int one, two, three = 1, 2, 3
string myName, int myAge = "Veronica", 34
Declare.dargon

You may have noticed that Dargon uses the newline character (enter key) as its statement delimiter (like ';' in a lot of programming languages). If you want a statement, such as a declaration, to extend multiple lines, you must add the tilde "~" character at the end of your line like so:

fixed int one, two, three = ~
1, ~
2, ~
3
